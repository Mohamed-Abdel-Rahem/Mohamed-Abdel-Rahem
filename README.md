## About

I build Flutter applications where every layer of the system has a deliberate reason to exist.

My work begins not at the widget tree, but at the domain boundary — defining contracts, isolating business logic, and ensuring that neither the UI nor the data layer can corrupt the core of the application. I treat mobile development as a discipline in **distributed systems design at the client level**: managing state machines, synchronization conflicts, offline-first persistence, and reactive data pipelines with the same rigor one would apply to a backend service.

I do not write features. I design systems that happen to have a user interface.

**Current focus:** Scalable offline-first architectures with real-time reconciliation using Firebase Firestore Streams and local SQLite persistence.

## Engineering Philosophy

> *"A codebase that cannot be understood by a new engineer in under an hour is a liability, not an asset. Complexity that serves no domain purpose is technical debt disguised as sophistication."*

Three principles guide every architectural decision I make:

**Isolation over convenience.** Business logic must never be aware of Flutter, Firebase, or any infrastructure detail. Use cases operate on abstractions; implementations are injected.

**Reactivity by contract.** State is a stream, not a snapshot. Every layer that consumes data should be prepared to handle emissions, errors, and loading states as first-class concerns.

**Failure is a feature.** Offline behavior is not an edge case. Every data operation is designed with network absence as the baseline assumption, not the exception.
